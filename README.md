# Seaborne Commodity Activity Monitor

Tracks seaborne commodity export/import **activity** in near-real-time by pairing free
**AIS vessel data** with a hand-curated **berth registry**. The berth a vessel calls at
supplies the commodity — a bulk carrier at a single-purpose copper terminal ≈ a copper
export. The tool is **commodity-agnostic**: coverage grows by adding rows to
`registry.json`, never by changing code.

> **This reports an activity *proxy*, not verified cargo tonnage.** That framing is a
> first-class constraint of the design, not a disclaimer bolted on at the end.

v1 seed = **Latin American copper + lithium** (10 terminals in Chile & Peru).

## Run it now (no keys required)

```bash
pip install -r collector/requirements.txt
python -m pytest collector/tests -q                       # 6 passing
python -m collector.collector --simulate --once --seed-history   # writes data/state.json
```

`--seed-history` backfills ~14 weeks of synthetic port calls so `activity_index` / `z`
are meaningful immediately, and nudges Puerto Coloso into a spike (`z≈3.7`) to exercise
the baseline maths. Run the streaming demo (writes a snapshot every ~45s) with
`--simulate --seed-history` (drop `--once`).

## View the console

```bash
cd web && python -m http.server 8000     # then open http://localhost:8000
```

The page boots from a bundled data snapshot (so opening `web/index.html` directly shows a
working map), then polls `web/data/*.json` every 20s for live updates. Terminals are
coloured by activity index (teal → amber → magenta); a terminal with |z| ≥ 2 emits a
**dashed magenta ring**, upgrading to a **solid ring** only once its berth polygon is
verified and the email alert arms. Click any terminal for metrics, a 30-day departures
sparkline, the customs destination split, and the latest export figure. The **Methodology**
button lays out every caveat from §8. To wire it to the live collector, point the collector's
`STATE_PATH` at `web/data/state.json` (and the jobs at `web/data/flows.json` / `prices.json`),
or set `DATA_BASE` at the top of the page to wherever those files are served. After changing
`_template.html`, re-run `python build.py`.

## Run it live

1. Get a free AISStream.io key (register via GitHub) and put it in `.env` as `AISSTREAM_KEY`.
   **The key stays server-side.** The frontend never touches AIS — it only polls the JSON files.
2. `AISSTREAM_KEY=... python -m collector.collector --live`

The bounding box is derived from the registry extents + margin (override with `BBOX_OVERRIDE`).

## Key decisions (all reversible via env / `registry.json`)

- **Working state = in-memory positions + SQLite event log.** Live vessel positions are
  ephemeral (in memory); the port-call event log lives in SQLite so baselines/z-scores are
  real and survive restarts. Fly.io config mounts a volume for it.
- **Verified-polygon alert gate (`REQUIRE_VERIFIED_POLYGON`, default on).** Anomaly *emails*
  fire only at `purity:high` terminals whose `berth_polygon` has been human-checked
  (`polygon_verified:true`). The seed polygons are unverified ~2 km placeholders, so **no seed
  terminal will email until you verify its polygon** — deliberate, given how hard the brief
  leans on polygon accuracy. Set `REQUIRE_VERIFIED_POLYGON=0` to arm alerts on placeholders
  (the simulator uses this to demo an alert firing). The `activity_index`/`z` numbers are
  always computed and published regardless of the gate.
- **AIS class is coarse by nature.** AIS ship-type can't distinguish bulk / container /
  general cargo (all "Cargo", codes 70–79); gas carriers ride in the tanker band. So commodity
  attribution uses the **berth tag as authoritative** and `vessel_class_hint` only as a
  fallback at multi-commodity berths.

---

## ⚠️ Coordinates need verification before you trust alerts

Per the brief's §9, the seed centroids are **approximate** and every `berth_polygon` is a
**first-pass ~2 km box**, flagged `polygon_verified:false`. Before relying on high-purity
signals or alerts, redraw each polygon *tightly around the actual loading berths* using the
World Port Index / a marine chart / satellite imagery, then set `polygon_verified:true`. A
loose polygon that swallows a whole harbour destroys port-call precision and inflates noise.

---

## The design invariant

**Commodity/region coverage expands by editing `registry.json` only** (terminals, and if
needed `commodity_taxonomy` strings + `country_meta` for a new producing country). If adding
a commodity or country ever requires touching code, the abstraction is wrong — fix the
abstraction, not the data. Everything downstream (detector, jobs, and the forthcoming
frontend filter) reads its domain knowledge from the registry — including the taxonomy-driven
commodity chips in the console.
